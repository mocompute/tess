# Tagged Union Variant Construction: Parser Internals

This document explains how the parser desugars tagged union variant construction
into the correct AST. It focuses on the **auto-wrapping** mechanism that makes
all construction forms return the tagged union type rather than the bare variant
struct type. It also documents parser pitfalls that have caused bugs.

## Background: Desugared Representation

A tagged union like:

```tl
Shape : | Circle { radius: Float } | Square { length: Float } | None
```

desugars into several C-level types:

| Generated type        | Purpose                                |
|-----------------------|----------------------------------------|
| `Shape`               | Struct with `tag` and `u` fields       |
| `__Shape__Tag_`       | Enum with `.Circle`, `.Square`, `.None`|
| `__Shape__Union_`     | C union of variant structs             |
| `Shape__Circle`       | Struct with `radius` field             |
| `Shape__Square`       | Struct with `length` field             |
| `Shape__None`         | Empty struct                           |

When the user writes `Circle(2.0)`, the parser must produce an AST equivalent to:

```
Shape(tag = __Shape__Tag_.Circle, u = __Shape__Union_(Circle = Shape__Circle(2.0)))
```

This wrapping is performed by `build_tagged_union_wrapping()`.

## Construction Forms and Their Code Paths

There are three distinct code paths in the parser that handle variant construction.
Understanding which path handles which syntax is critical — adding auto-wrap logic
in the wrong path is a recurring source of bugs.

### Path 1: Unscoped construction — `toplevel_tagged_union`

Handles: `Circle(2.0)`, `Circle(radius = 2.0)`, `None()`, bare `None`

When the parser processes a tagged union definition, `toplevel_tagged_union()`
generates **constructor functions** for each variant (line ~3670). These are
ordinary functions whose bodies call `build_tagged_union_wrapping()` to produce
the tagged union struct. For example, `Circle(2.0)` calls a generated function:

```c
// Generated by toplevel_tagged_union:
Circle__1(radius) { Shape(tag = __Shape__Tag_.Circle, u = ...) }
```

The special case of bare `None` (no parentheses) is handled in `a_value()` at
line ~1416: any bare `None` symbol is promoted to `ast_node_create_nfa_tc(None, {}, {})`,
turning it into a zero-arg type constructor call that resolves to the generated
`None__0()` function.

**No auto-wrap logic needed in this path** — the generated functions already
return the tagged union type.

### Path 2: Scoped construction — `maybe_mangle_binop` (nested-type branch)

Handles: `Shape.Circle(2.0)`, `Shape.Circle(radius = 2.0)`, `Shape.None()`,
`Shape.A` (bare symbol), `Foo.Shape.Circle(...)`, `Foo.Shape.A`

When the parser encounters a dot operator in `parse_expression()`, it calls
`maybe_mangle_binop()`. The nested-type branch (line ~1826) fires when:

1. The operator is `.`
2. The left side is a symbol found in `nested_type_parents`
3. The right side is a symbol or NFA node whose mangled name exists in the
   module's symbol table

This branch rewrites `Shape.Circle(2.0)` to `Shape__Circle(2.0)` (name mangling),
then wraps the result via `build_tagged_union_wrapping()` if the parent is in
`tagged_union_variant_parents`.

**Two sub-cases exist for the right-hand side:**

- **NFA node** (e.g., `Shape.Circle(2.0)`): The right side is already parsed as
  a function call. The wrapping uses it directly as the inner call.

- **Bare symbol** (e.g., `Shape.A` or `Op.A`): The right side is a plain symbol
  node, not an NFA. It must be **promoted to a zero-arg NFA_TC** before wrapping —
  the same pattern used for bare `None` sugar. Without this promotion, the symbol
  passes through unwrapped and produces the variant struct type (`Op__A`) instead
  of the tagged union type (`Op`).

### Path 3: Type constructor fallback — `a_type_constructor`

Handles: `Circle(radius = 2.0)` when `a_funcall` fails

This is NOT a tagged-union-specific path. It's the general type constructor parser
that handles `Name(field = value, ...)` syntax. It fires as a fallback when
`a_funcall` fails to parse the arguments (see "Why `a_funcall` never sees `=`"
below).

For tagged union variants, the unscoped constructor functions generated by Path 1
handle this — `a_type_constructor` parses the syntax and the generated function
provides the wrapping.

## Critical Parser Invariant: `a_funcall` Never Sees `=` Arguments

**This is the most important thing to understand about tagged union construction
parsing, and the source of a prior dead-code bug.**

In `a_value()`, `a_funcall` is tried **before** `a_type_constructor`:

```c
static int a_value(parser *self) {
    if (0 == a_try(self, a_funcall)) return 0;        // tried first
    if (0 == a_try(self, a_type_constructor)) return 0; // fallback
    ...
}
```

Both start the same way: parse an identifier, then open parenthesis. But they
diverge on argument parsing:

| Parser             | Argument parser        | Handles `=`? |
|--------------------|------------------------|--------------|
| `a_funcall`        | `a_expression`         | No*          |
| `a_type_constructor` | `a_field_assignment` | Yes          |

*`a_expression` calls `parse_expression()` which calls `a_binary_operator()`.
`a_binary_operator()` **explicitly rejects `tok_equal_sign`** (line ~783):

```c
case tok_equal_sign:
    return 1;  // reject — not a binary operator
```

This means when parsing `Circle(radius = 2.0)`:

1. `a_funcall` starts: parses `Circle`, `(`, then tries `a_expression` on `radius = 2.0`
2. `a_expression` parses `radius` as a base expression
3. The Pratt parser loop tries `a_binary_operator` on `=` — it's rejected
4. `parse_expression` returns just `radius`
5. `a_funcall` now expects `)` or `,` — sees `=` instead — **fails and backtracks**
6. `a_type_constructor` takes over: parses `Circle`, `(`, then `a_field_assignment`
   which correctly handles `radius = 2.0`

**Consequence:** Any auto-wrap code placed inside `a_funcall` that checks for `=`
arguments is dead code. The `=` token causes `a_funcall` to fail before reaching
any such check.

### Historical Bug: Dead Code in `a_funcall` (commit 310345a9, removed cc08bc21)

A previous implementation added a `variant_to_tagged_union` hashmap (mapping
variant names to their parent tagged union names) and a 43-line auto-wrap block
inside `a_funcall` that checked whether the first argument was a binary op with
`=`. This code was unreachable:

```c
// DEAD CODE — was in a_funcall after argument parsing:
if (args.size && ast_node_is_binary_op(args.v[0])) {
    str *tu_name_ptr = str_map_get(self->variant_to_tagged_union, original_name);
    if (tu_name_ptr && 0 == str_cmp_c(args.v[0]->binary_op.op->symbol.name, "=")) {
        // ... wrapping logic ...
    }
}
```

This was dead because:
- If arguments contain `=`, `a_funcall` fails before reaching `done:` (backtrack)
- If arguments don't contain `=`, the `ast_node_is_binary_op` + `"="` check fails

The `variant_to_tagged_union` map was **only read** by this dead block. It also
had a latent cross-module collision bug (different modules could have variants
with the same name, overwriting each other in the map), but that was moot since
the map was never read on any reachable path.

**Lesson:** Before adding auto-wrap or desugaring logic to a parser function,
trace the actual token flow to verify the code is reachable. Pay special attention
to which tokens cause `a_try()` backtracking.

## Parser Data Structures for Tagged Unions

| Structure                        | Type     | Purpose |
|----------------------------------|----------|---------|
| `nested_type_parents`            | str hset | All types that have nested types (tagged unions, nested structs). Used by `maybe_mangle_binop` to recognize `Parent.Child` dot syntax. |
| `tagged_union_variant_parents`   | str hset | Subset of `nested_type_parents`: only tagged union parent names. Used to decide whether to auto-wrap in `maybe_mangle_binop`. |

Both are populated by `toplevel_tagged_union()` during definition parsing.

## Summary of Auto-Wrap Locations

| Syntax                       | Parser function            | Wrap mechanism                    |
|------------------------------|----------------------------|-----------------------------------|
| `Circle(2.0)`               | Generated constructor fn   | Function body calls `build_tagged_union_wrapping` |
| `Circle(radius = 2.0)`      | Generated constructor fn   | Function body calls `build_tagged_union_wrapping` |
| `None()` or bare `None`     | Generated constructor fn   | Function body calls `build_tagged_union_wrapping` |
| `Shape.Circle(2.0)`         | `maybe_mangle_binop`       | NFA branch wraps directly         |
| `Shape.Circle(radius = 2.0)`| `maybe_mangle_binop`       | NFA branch wraps directly         |
| `Shape.A` (bare symbol)     | `maybe_mangle_binop`       | Promote to NFA_TC, then wrap      |
| `Foo.Shape.Circle(...)`     | `maybe_mangle_binop` (2x)  | First dot resolves module, second dot hits nested-type branch |
| `Foo.Circle(...)`           | Module branch of `maybe_mangle_binop` | Resolves to generated constructor fn |
